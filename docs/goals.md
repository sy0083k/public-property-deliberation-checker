## 1. 서비스 목표
- 공유재산 관련 안건이 `심의`, `심의 비대상`, `심의 + 관리계획 수립`, `심의 + 관리계획 변경 수립`, `심의/관리계획 제외` 중 어디에 해당하는지 일관되게 판별한다.
- 담당자가 원문 항목, 예외 사유, 취득·처분 수치를 입력하면 동일 기준으로 결과를 재현할 수 있게 한다.
- 판정 결과와 단계별 근거를 감사 이력으로 남겨 사후 검토를 가능하게 한다.
- 서산시 기준 규칙을 기본값으로 제공하되, 규칙 프로파일 확장을 위한 구조를 유지한다.

## 2. 사용자 경험 목표
- 내부 사용자가 별도 교육 없이 입력, 판정, 결과서 출력까지 한 화면에서 처리할 수 있게 한다.
- 사례 유형과 관련 법령을 함께 제시해 입력 오류와 해석 혼선을 줄인다.
- 결과 문서에 최종 결정, 사유, 입력 요약, 관련 법령, 단계별 판정 흐름이 누락되지 않게 한다.
- 이전 판정 건을 날짜/부서/결정 기준으로 빠르게 조회할 수 있게 한다.

## 3. 데이터 정확성 목표
- 사례 생성 입력값은 API 경계에서 길이와 필수값을 검증한다.
- 답변 저장은 동일 `case_id + question_code` 조합에 대해 멱등적으로 갱신된다.
- 규칙 카탈로그 로딩 실패나 스키마 불일치가 있으면 앱 시작 시 명시적으로 실패한다.
- 활성 규칙 프로파일은 카탈로그 임계값과 동기화돼야 한다.

## 4. 운영성/유지보수 목표
- 판정 로직은 `backend/app/rules.py`에 집중해 회귀 테스트를 쉽게 유지한다.
- 규칙 원본은 YAML 카탈로그를 기준으로 관리한다.
- 기본 운영 저장소는 SQLite로 유지하되, 마이그레이션과 스키마 검증 실패를 조기에 감지한다.
- 문서, 테스트, 앱 표시명이 동일한 제품 정체성을 유지하도록 관리한다.

## 5. 보안 목표
- 내부용 앱으로서 외부 공개용 인증/권한 체계는 범위 밖으로 두되, CORS와 환경변수 기반 설정을 유지한다.
- 비밀값은 코드와 문서에 하드코딩하지 않는다.
- API 응답에는 공개 스키마에 정의된 필드만 노출한다.

## 6. 품질 목표
- 규칙 분기와 임계값 회귀를 자동 테스트로 유지한다.
- 사례 생성, 답변 저장, 평가, 조회, 목록 API 흐름이 통합 테스트로 검증된다.
- 프런트엔드 빌드는 타입 오류 없이 완료되어야 한다.

## 7. 비목표
- 다중 사용자 권한 체계와 승인 워크플로는 현재 범위 밖이다.
- 실시간 협업 편집, 외부 로그인, 알림 시스템은 현재 범위 밖이다.
- 지리정보 지도 시각화, 업로드형 배치 처리, 외부 API 기반 보강 작업은 현재 범위 밖이다.
